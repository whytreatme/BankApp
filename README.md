## 开发历程

### 事故编号：20260228-02

**问题描述：**
在进行 500 机器人压力测试时，客户端出现 WinError 10061，服务器主动拒绝连接。

**原因分析：**
- 原有架构为单线程 Reactor，数据库查询等耗时操作阻塞了事件循环。
- 操作系统 Backlog 队列满，导致 accept 无法及时处理新连接。

**解决过程：**
1. 方案一：增加 Backlog 大小（治标不治本）。
2. 方案二：客户端限流（非真实应用场景）。
3. 方案三（最终采纳）：引入线程池。主线程只负责 I/O，业务逻辑下发至 WorkerTask 异步执行。

**技术难点：**
- 处理 Qt 中 QSqlDatabase 的线程亲和性问题，实现了基于线程 ID 的连接池。   



### 事故编号：20260316-02
**问题描述**：
在进行 500 并发压力测试时，客户端出现大量 `WinError 10054` 错误，服务端控制台频繁打印“余额查询失败”，随后服务端进程异常崩溃（Terminated Abnormally）。

**原因分析**：
1. **数据库连接瓶颈**：由于采用“一线程一连接”模式，500 个并发线程尝试同时建立 MySQL 连接，超过了 MySQL 默认的最大连接数限制（通常为 151），导致后续线程获取数据库句柄失败。
2. **异步对象生命周期冲突（野指针）**：在 `WorkTask` 执行耗时数据库操作期间，若客户端主动断开连接，I/O 线程会销毁 `QTcpSocket` 对象。当 `WorkTask` 完成并尝试通过信号回传结果时，访问了已销毁的 `socket` 指针，触发非法内存访问导致崩溃。

**解决方案**：
- [ ] **资源限流**：将 `QThreadPool` 的最大线程数限制为 50，通过任务队列缓冲并发请求，保护后端数据库。
- [ ] **防御式指针管理**：在 `WorkTask` 中引入 `QPointer<QTcpSocket>`（弱指针），在回调前校验 Socket 的合法性，彻底杜绝野指针崩溃。
- [ ] **数据库配置优化**：计划调高 MySQL 的 `max_connections` 参数。
